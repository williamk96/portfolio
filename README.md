# CS250 Project Reflection

## Interpreting User Needs and Implementing Them Into a Program

Interpreting user needs begins with understanding the problem the user is trying to solve rather than immediately thinking about code. Instead of asking “What features should I build?”, I focus on “What outcome is the user trying to achieve?” This requires clarifying requirements, identifying constraints, and translating high-level goals into actionable functionality.

Creating **user stories** helps bridge the gap between business needs and technical implementation. A well-written user story follows the structure:

> As a [user], I want [functionality], so that [benefit].

This format forces clarity in three important areas:

- Who the feature is for
- What behavior is expected
- Why it matters

The “why” is especially important. It prevents building unnecessary functionality and keeps development aligned with value delivery. User stories also make requirements testable. If acceptance criteria are clearly defined, I can validate whether the implementation truly satisfies the need.

User stories essentially act as a contract between stakeholders and developers. They convert abstract ideas into concrete, measurable development goals.

## My Approach to Developing Programs and Agile Processes I Plan to Use

When developing a program, I start by breaking the problem into manageable components. I identify core functionality first (the minimum viable product), then build incrementally. I prefer to:

1. Define requirements clearly
2. Design the architecture at a high level
3. Implement small, testable units
4. Test continuously
5. Refactor when necessary

Rather than attempting to build everything at once, I focus on iterative improvement. This reduces risk and allows adjustments based on feedback.

In future development work, I plan to incorporate Agile practices such as:

- **Sprint Planning** to define short-term development goals
- **Daily Stand-ups** (even if informal) to track progress and blockers
- **Backlog Refinement** to continuously clarify requirements
- **Sprint Reviews** to validate work with stakeholders
- **Retrospectives** to improve team performance

I also hope to incorporate version control discipline (Git branching strategies), automated testing where possible, and continuous integration practices. Agile is not just about speed, it is about adaptability and consistent delivery of value.

## What It Means to Be a Good Team Member in Software Development

Being a good team member in software development means prioritizing the success of the team over individual recognition. It requires:

- Clear communication
- Accountability
- Willingness to receive and give constructive feedback
- Transparency about progress and blockers
- Respect for diverse perspectives

A strong team member does not work in isolation. They document their work clearly, write readable code, and consider how their decisions affect integration with others’ contributions.

It also means being proactive. If something is unclear, ask. If there is a risk, raise it early. If a teammate is blocked and you can help, step in.

Technical skill matters, but reliability, collaboration, and professionalism often determine the success of a development effort.
